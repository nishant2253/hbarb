Smart Contracts — Native Hedera + HSCS
ContractCreateTransaction via SDK · HFS bytecode storage · Hardhat as secondary option
Phase Summary
This phase deploys AgentRegistry.sol using Hedera's NATIVE contract deployment — ContractCreateTransaction
via the Hedera SDK — rather than Hardhat as the primary path. Bytecode is first stored in HFS (Hedera File
Service) as an on-chain file, then deployed from it. Hardhat is still available as a secondary option via JSON-RPC
relay.
Steps in This Phase:
1. Write AgentRegistry.sol (Solidity 0.8.24 + OpenZeppelin 5.x)
2. Compile with Hardhat (compilation only — deployment is native)
3. Store compiled bytecode in HFS using FileCreateTransaction
4. Deploy contract from HFS file using ContractCreateTransaction (native Hedera)
5. Verify contract on HashScan
6. Save contract ID + EVM address to .env
n PRIMARY deployment is now via Hedera SDK ContractCreateTransaction — storing bytecode in HFS first.
This uses 5 Hedera services at once (HFS + HSCS) and feels 100% Hedera-native vs
eth_sendRawTransaction.

4.1 Hedera Consensus Service (HCS) — The Audit Heart Color(0,.662745,.729412,1)
n HCS is the most critical Hedera service for TradeAgent. It is a decentralized, aBFT-ordered message
sequencing service — a public notary that stamps every message with a timestamp that is mathematically
impossible to forge or alter.
What HCS Actually Is
– HCS does not store message content on consensus nodes — it records the ORDER and TIMESTAMP using
Hashgraph's virtual voting algorithm
– Messages are submitted to a Topic — a logical channel identified by a topic ID (e.g. 0.0.4823901)
– Every submitted message gets a consensus_timestamp — the median of when all network nodes received
the message, determined by aBFT virtual voting
– The consensus timestamp is final in 3–5 seconds and guaranteed correct even if up to 1/3 of nodes are
malicious or offline
– Full message history stored and queryable for free via Mirror Node REST API and gRPC streaming
How TradeAgent Uses HCS
– Every deployed agent gets its own dedicated HCS Topic created at deployment time (via
TopicCreateTransaction)
– Before any trade executes, the full decision (signal, price, indicators, confidence, reasoning) is submitted to
the agent's topic
– The aBFT timestamp proves the agent made this decision at this moment — before the outcome was
known
– Anyone — trader, regulator, marketplace buyer — can query Mirror Node and verify the complete, unaltered
decision history
– Performance statistics on TradeAgent are not numbers on a screen — they are cryptographically verifiable
facts
– HCS-10 additionally registers each agent in the Hedera ecosystem for inter-agent communication
n Each HCS message costs $0.0001. An agent firing 20 decisions/day costs $0.73/year in Hedera fees —
making continuous audit logging economically viable at retail scale. On Ethereum this would cost $50–$500/year
at average gas prices.
